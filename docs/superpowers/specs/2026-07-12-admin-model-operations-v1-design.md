# 管理员模型运营中心 v1 设计

## 目标

在现有管理员审计中心增加只读模型运营视图，让管理员可以查看当前及历史模型的目录信息、已持久化使用量、Agent 运行情况和最近一次已导入压测，并从模型详情跳转到该模型的对话列表。

## 数据边界

- LiteLLM 继续作为当前模型目录的唯一事实源，Fusion 不增加模型增删改能力。
- 当前模型只读取经过 `litellm_catalog` 剥离后的白名单字段，禁止返回 `litellm_params`、API key、API base、底层模型名或原始 metadata。
- 历史模型由 `Conversation.model_id`、`Message.model_id`、`AgentSession.model_id` 和 `PerformanceRun.model_id` 等已持久化记录补齐。目录可用且不存在时标记为 `historical`；目录降级时无法确认的模型标记为 `unknown`。
- 会话数表示当前选择该模型的会话；用户数表示这些会话的去重用户。
- 回复数与 Token 只统计已持久化且带实际 `Message.model_id` 的助手消息。该口径不是计费总账，也不覆盖失败请求、未持久化的辅助调用或历史上被覆盖的续写用量。
- Agent 错误数只来自 `AgentSession.status=error`。Agent 总耗时是编排墙钟时间，不能作为模型延迟。
- 最近压测来自管理员已导入的 `PerformanceRun`，不能解释为实时线上成功率或延迟。
- v1 不下发价格。现有目录价格只有币种，没有每 Token、每千 Token 或每百万 Token 的计价基准；待目录补齐明确 basis 后再纳入运营展示。

## 接口与交互

- `GET /api/admin/audit/models`：分页、搜索并返回当前与历史模型的安全摘要。
- `GET /api/admin/audit/models/{model_id}`：返回单模型安全详情；支持包含 `/` 的模型 ID。
- 两个接口沿用管理员审计权限，并分别记录模型列表和模型详情访问事件。
- 管理页增加“模型”Tab；详情 ID、对话筛选模型 ID 都写入 URL，支持刷新恢复、浏览器返回和手势返回。
- “查看该模型的对话”跳转到对话 Tab，并同步对话筛选控件。

## 验收矩阵

| 编号 | 场景 | 验收点 |
|---|---|---|
| AMO-01 | 当前目录模型 | 展示白名单目录字段和正确聚合数据 |
| AMO-02 | 历史模型 | 目录缺失仍可列出和查看详情，明确标记历史 |
| AMO-03 | 模型 ID 含 `/` | 详情路由、前端编码和深链接均可用 |
| AMO-04 | 目录暂时为空 | 已有历史数据不消失，页面不泄露底层错误 |
| AMO-05 | 无使用记录 | 当前模型以零值安全展示 |
| AMO-06 | 分页与筛选 | 搜索、提供商、目录状态筛选结果与总数一致 |
| AMO-07 | 权限与审计 | 非管理员返回 403；列表与详情产生安全审计事件 |
| AMO-08 | 安全投影 | 响应不含凭据、原始配置、Agent 错误正文或伪延迟字段 |
| AMO-09 | 查询规模 | 聚合查询数量不随模型数量线性增长 |
| AMO-10 | URL 历史 | 打开详情、刷新、关闭、返回、模型到对话联动均保持路由一致 |
| AMO-11 | 加载与失败 | 列表/详情 loading、空态、错误和重试可见且无旧请求覆盖 |
| AMO-12 | 生产回归 | 真实管理员登录态验证列表、详情、历史返回、对话联动和 console |
