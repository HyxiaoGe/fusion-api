# 自主联网判断与执行过程修复计划

## 背景

本轮问题包含两个用户可见缺陷：

1. 简单问候、稳定常识等不需要联网的问题，可能仍触发固定的搜索/读取计划。
2. 已发生的工具过程展示不够精确：成功读取、跳过不可读网页、搜索关键词与 URL 文本边界容易混淆。

其中第 1 点是 P0：真实回归不能显式要求“联网搜索”，必须用自然问题验证模型是否自行判断。

## 用例矩阵

| 编号 | 场景 | 输入 | 预期 |
| --- | --- | --- | --- |
| P0-A | 自然问题应自主搜索 | `微信A2A互通怎么用？` | 模型主动调用 `web_search`，必要时深读高价值来源 |
| P0-B | 自然问题应自主搜索 | `OpenAI 最近发布了哪些产品更新？` | 模型主动调用 `web_search`，搜索关键词带当前年份或时间限定 |
| P0-C | 自然问题不应搜索 | `你好，你是谁？` | 不调用 `web_search`，不展示搜索/读取计划 |
| P0-D | 自然问题不应搜索 | `1+1等于几？` | 不调用 `web_search`，直接回答 |
| UI-A | 读取部分失败 | 1 个成功读取 + 1 个不可读网页 | 执行摘要展示成功读取和跳过数量，不暴露 `url_read` / `reader-service` |
| UI-B | 思考正文裸 URL | `https://example.com/a，原因是...` | 链接只包含 URL，中文说明保留为普通文本 |

## 实施

1. 后端先补测试，锁住工具可用时注入“自主联网判断”规则，规则同时覆盖应搜索和不应搜索示例。
2. 后端补强 `web_search` 描述，明确不依赖用户是否说“搜索/联网”，但问候、数学、翻译、代码、创意写作不强制搜索。
3. 前端先补测试，锁住执行摘要的成功读取/跳过数量和 URL 边界。
4. 前端修正执行过程模型和 Markdown 预处理。
5. 运行后端/前端针对性测试，必要时再跑更大范围测试。
6. 合并部署后，复用已登录 Chrome 现有标签做真实生产回归；记录新建对话 URL、输入、预期、实际、console error、刷新结果和结论。
