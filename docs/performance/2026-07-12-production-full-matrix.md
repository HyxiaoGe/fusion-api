## 方法与边界

L1 使用 runner 通过生产公网域名发起冷连接请求，覆盖 Cloudflare、Nginx、Auth/Fusion 服务及数据库；它是偏保守的公网端到端数据，不等价于浏览器 HTTP/2 连接复用后的页面体验。L2-L4 使用真实生产 SSE、真实模型生成、真实 Redis Stream 恢复和停止持久化路径。每条流的提示词包含进程内唯一标识，避免命中上游 prompt cache。

资源门禁在每个 L1-L3 档位后执行，并在 L4 每分钟执行。门禁监控容器启动时间、OOM、API 内存、PostgreSQL 连接、Redis 拒绝连接/淘汰键和主机可用内存；Prometheus 缺样本时仍 fail-closed。

## 压测过程暴露并已修复的问题

1. 同一新用户首次并发请求可能竞争插入 `social_accounts`，触发唯一约束 500；已增加精确约束恢复与身份一致性校验。
2. Windows CI 使用 `unittest discover`，压测测试不应强依赖 pytest；已兼容。
3. Prometheus CPU `rate[2m]` 在容器刚部署时可能短暂缺样本；已改为 5 分钟窗口、有限指数退避，耗尽仍 fail-closed。
4. 相同提示词会命中上游缓存，使 L2 指标失真、L3 无法真实断开；已为每条会话注入唯一压测标识。
5. L4 将无参数资源门禁直接作为窗口回调，触发类型错误；已增加回调适配，并规范化异常停止码。
